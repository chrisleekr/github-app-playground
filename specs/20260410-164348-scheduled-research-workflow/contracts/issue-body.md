# Contract: Research-Issue Body

**Feature**: Scheduled Research Workflow
**Branch**: `20260410-164348-scheduled-research-workflow`

## What this contract covers

The exact markdown structure that every research-filed issue MUST contain. This is the **output contract** of the workflow — the document the maintainer reads on triage day.

The body is enforced **by the agent's prompt** (the prompt template embedded in `.github/workflows/research.yml`), not by a separate validator. The agent MUST write the issue body to `/tmp/issue-body.md` and then call `gh issue create --body-file /tmp/issue-body.md`. Embedding the body directly in the `gh issue create --body` argument is forbidden because it bypasses Markdown escaping for code fences and Mermaid diagrams.

---

## Body sections (mandatory order, mandatory content)

````markdown
## Finding

[1–3 paragraphs of plain prose. Describes the discovery. Cites exact file paths and (optionally) line numbers. Explains current behaviour and how it can be improved.]

## Diagram

```mermaid
[A Mermaid diagram that EXPLAINS the finding. See "Mermaid diagram rules" below. Must be wrapped in a triple-backtick fenced block tagged `mermaid`.]
```
````

## Rationale

[1–3 paragraphs explaining why the finding matters. Includes metrics or quantitative estimates where possible (e.g., "~200ms p99 latency", "~30% test coverage gap", "blocks N PRs/week").]

## References

- **Internal**:
  - `path/to/file.ts:42` — [one-line context]
  - `path/to/other.ts` — [one-line context]
  - …
- **External**:
  - https://… — [one-line context]
  - https://… — [one-line context]
  - …

## Suggested Next Steps

1. [First concrete action]
2. [Second concrete action, or alternative if applicable]
3. [Optional further actions]

## Areas Evaluated

[What the agent looked at this run. Includes the chosen focus area and any ancillary subsystems read for context. Single bullet list, free-form.]

---

_Generated by scheduled research workflow run #<runId> on <YYYY-MM-DD>_

````

---

## Mandatory section list (testable from issue body alone)

| # | Section | Heading text | Content rule |
|---|---|---|---|
| 1 | Finding | `## Finding` | Non-empty. ≥1 paragraph. ≥1 internal file path cited. |
| 2 | Diagram | `## Diagram` | Non-empty. Contains exactly one ` ```mermaid ` fenced code block. |
| 3 | Rationale | `## Rationale` | Non-empty. ≥1 paragraph. |
| 4 | References | `## References` | Non-empty. Contains both `**Internal**:` and `**External**:` sub-bullets. ≥1 entry under each. |
| 5 | Suggested Next Steps | `## Suggested Next Steps` | Non-empty. Numbered list with ≥1 item. |
| 6 | Areas Evaluated | `## Areas Evaluated` | Non-empty. Mentions the run's `selectedFocusArea` at minimum. |
| 7 | Footer | Italicised line at end | Format `*Generated by scheduled research workflow run #<runId> on <YYYY-MM-DD>*`. |

If any of (1)–(6) is missing or empty, the issue is malformed and the agent's quality gate MUST have prevented the issue from being filed (i.e., the run's outcome should have been `nothing_qualified`, not `issue_created`). A malformed issue in production indicates a quality-gate bug that should be fixed in the prompt template.

## Mermaid diagram rules

These rules are inherited from the user's global rules (`~/.claude/CLAUDE.md`) for GitHub Mermaid renderer compatibility, plus additional rules from the reference workflow's empirical experience:

| Rule | Reason | Example violation |
|---|---|---|
| Use `<br/>` for line breaks, never literal `\n` | GitHub renderer-specific | `node["line1\nline2"]` ❌ |
| Avoid parentheses inside node labels | They break Mermaid's parser | `node["foo(bar)"]` ❌ |
| Use `:::className` inline syntax with `classDef` | Separate `class NodeId className` statements fail in GitHub's renderer | `class A keep` ❌ → `A["A"]:::keep` ✅ |
| `classDef` properties separated by **semicolons**, not commas | Comma-separated `classDef foo fill:#196f3d,color:#ffffff` is broken in GitHub's renderer (per the reference's empirical evidence) | `classDef foo fill:#196f3d,color:#ffffff` ❌ → `classDef foo fill:#196f3d;color:#ffffff` ✅ |
| Avoid multiple `subgraph` blocks; flatten or use one | GitHub renderer fails when a second `subgraph` follows `end` | Two `subgraph` blocks ❌ |
| Node IDs ≥3 characters | Reserved-word collisions | `a` ❌ → `nodeA` ✅ |
| `classDef` colour pairs MUST meet WCAG 2 AA (≥4.5:1 contrast) | Per user's global rules — accessibility | Light grey on white ❌ |
| Diagram MUST explain the finding, not decorate it | Per user's global rules and the reference | A 50-node "architecture map" with no relation to the finding ❌ |
| Diagram MUST be inside a fenced ` ```mermaid ` block | Required by GitHub Markdown to render | Plain text Mermaid syntax ❌ |
| Prefer simple `graph LR` or `graph TD` over `stateDiagram-v2` / `gitGraph` | Most reliably rendered by GitHub | Custom diagram types ❌ |

## Title contract (companion to body)

The issue title is set by `gh issue create --title` and MUST follow:

```text
<type>(<area>): <summary>
````

| Field       | Constraint                                                                                                                                                                  |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `<type>`    | One of `fix`, `feat`, `perf`, `refactor`, `security`, `test`, `docs`, `chore`, `build`, `ci`. Matches `.github/labeler.yml`. The agent picks based on the finding's nature. |
| `<area>`    | One of the 10 focus areas in `data-model.md` §5 (without the `area: ` label prefix). Always equals `selectedFocusArea` for the run that produces the issue.                 |
| `<summary>` | Free text. ≤120 characters. Imperative mood preferred ("add X", "fix Y").                                                                                                   |

**Examples** (from the reference workflow's example list, adapted to this repo's focus areas):

- `fix(idempotency): tracking-comment lookup misses runs that crashed before posting the comment`
- `feat(observability): emit per-request token-cost histogram via pino child logger`
- `perf(pipeline): cache git-clone in a per-installation work-tree pool`
- `security(mcp): prompt-injection via user-supplied PR body when constructing inline-review markdown`
- `test(webhook): no test for the concurrent-trigger rejection path`
- `docs(agent-sdk): missing JSDoc on prompt-builder's exported public API`

## Label contract (companion to title and body)

Every issue created by the workflow MUST be filed with **exactly two** labels via a single `gh issue create --label "research,area: <focusArea>"` invocation. This is enforced by the agent's prompt (the prompt instructs the agent to use that exact `--label` argument). The labels are:

| Label               | Purpose                                            | Constraint                                                                                                                                                                              |
| ------------------- | -------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `research`          | Marker — identifies the issue as workflow-produced | Exactly this string. Created by `gh label create research --description "..." --color 0e8a16 --force` immediately before issue creation.                                                |
| `area: <focusArea>` | Subsystem the finding pertains to                  | Exactly `area: ` + the focus area name (with the space). Created by `gh label create "area: <focusArea>" --description "..." --color 1d76db --force` immediately before issue creation. |

If either label creation step fails, the issue creation step MUST also fail. (The prompt instructs the agent to chain the commands so a label-creation failure aborts before issue-creation.)
