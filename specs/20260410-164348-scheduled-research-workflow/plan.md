# Implementation Plan: Scheduled Research Workflow

**Branch**: `20260410-164348-scheduled-research-workflow` | **Date**: 2026-04-10 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/20260410-164348-scheduled-research-workflow/spec.md`

## Summary

Add a single GitHub Actions workflow at `.github/workflows/research.yml` that runs once every 24 hours (and on-demand via `workflow_dispatch`) to invoke `anthropics/claude-code-action@v1` against this repository, restricted to a single focus area per run, and produces **at most one** deeply researched, semantically de-duplicated GitHub issue per run. The workflow enforces a hard 1-hour wall-clock budget per run, fails fast when required secrets are missing, never modifies repository content (no commits, branches, PRs, or tags), and surfaces technical failures via GitHub Actions' built-in workflow-failure notifications. The feature ships as **infrastructure-as-config only** — it does **not** touch any code in `src/`, does **not** add runtime dependencies, and does **not** modify the existing `@chrisleekr-bot` webhook server.

## Technical Context

**Language/Version**: YAML (GitHub Actions workflow syntax) + Markdown (prompt template). No TypeScript, no Bun runtime code.
**Primary Dependencies**: `anthropics/claude-code-action@v1` (pinned-major action), `actions/checkout@v6` (already used elsewhere in this repo), `gh` CLI (preinstalled on `ubuntu-latest`). No new `package.json` dependencies.
**Storage**: None. All state lives in GitHub (issues, labels, run history). No database, no cache, no temp files persisted between runs.
**Testing**: Workflow validation via `actionlint` (run locally and in `bun run check` if integrated) plus a documented manual `quickstart.md` smoke test (single `workflow_dispatch` invocation on this branch). `bun test` does **not** apply — there is no TypeScript code to test. See "Constitution Check → Principle V" below for the rationale.
**Target Platform**: GitHub-hosted runner `ubuntu-latest`. No self-hosted runners, no matrix.
**Project Type**: Infrastructure-as-config (single CI/CD workflow file). Not "library", not "web-service", not "CLI" — closest analogue in this repo is `.github/workflows/semantic-release.yml`.
**Performance Goals**: Single run ≤60 minutes wall-clock (hard ceiling, FR-005). Typical successful run is expected to complete well under that ceiling.
**Constraints**:

- ≤1 issue created per run (FR-015).
- 0 commits/branches/PRs/tags created per run (FR-007, SC-004).
- 0 concurrent runs (FR-004) — enforced via GitHub Actions `concurrency` block with `cancel-in-progress: false`.
- 0 dependence on the application's own secrets (the workflow runs in CI and uses Actions secrets; it does **not** reuse the webhook server's GitHub App private key).
  **Scale/Scope**: 1 run/day automatic + occasional manual dispatch ≈ 30–35 runs/month. Approximately 7 issues/week of triage volume on the optimistic upper bound.
