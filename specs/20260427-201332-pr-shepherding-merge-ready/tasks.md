## Phase 3: User Story 1 — Drive a stuck PR to merge-ready (Priority: P1) 🎯 MVP

**Goal**: A maintainer issues `bot:ship` on a PR whose only blockers are mechanical (lint, format, narrow review-comment edits, rebase). The bot drives the PR to a verified `MergeReadiness.ready === true` state without holding daemon slots, surviving restarts, and resolving review threads it has addressed.

**Independent Test** (per spec User Story 1): Open a PR with a deliberate lint failure and one review comment requesting a renamed variable. Trigger `bot:ship`. Verify the bot pushes a fix commit, replies + resolves the review thread, waits for CI green, and updates the tracking comment to merge-ready — without human keystrokes between trigger and merge-ready.

## Implementation Strategy

### MVP First (US1 + US2 — both P1)

The spec splits the P1 priority across two stories because halting safely is part of being correct. US1 alone (drive to ready) without US2 (halt safely) is dangerous in production — a probe-driven `bot:ship` that doesn't recognise design-discussion threads will push junk fixes. So the MVP is **US1 + US2 together**.

1. Phase 1: Setup
2. Phase 2: Foundational
3. Phase 3: US1 (drive to ready, includes restart-safety + reactor)
4. Phase 4: US2 (halt safely, includes signature ledger + reviewer barrier)
5. **STOP and VALIDATE**: dogfood the local-e2e flow (`quickstart.md` §"P1 Acceptance"); fix issues before US3/US4
6. Deploy with feature flags off; flip them on for one PR at a time

### Incremental Delivery After MVP

7. US3 — better visibility + restart-safety hardening — ship-and-soak
8. US4 — abort safety valve — ship-and-soak
9. Polish phase: docs + perf + flag removal in a follow-up PR

After Phase 2 completes:

- Developer A: US1 (largest phase; the architecture payload)

- Developer C: US3 docs/observability + Polish (T061–T066 can start any time once intent + probe modules exist)
